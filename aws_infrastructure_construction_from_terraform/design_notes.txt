設計メモ

このプロジェクトは、シンプルなFastAPIアプリケーションを開発し、その実行環境（インフラストラクチャ）の構築からアプリケーションのコンテナ化、デプロイまでの一連のプロセスを、Infrastructure as Code (IaC) とCI/CDパイプラインを用いて自動化するものである。

1. ライブラリ・ツールの選定理由
本プロジェクトでは、役割に応じて以下の主要なツールとPythonライブラリを使用している。

アプリケーション開発
FastAPI, Uvicorn:
用途: Web APIサーバーを構築し、実行するためのフレームワークとASGIサーバー。
選定理由: FastAPIはPythonの型ヒントを活用したモダンで高性能なフレームワークであり、APIのドキュメンテーションを自動生成する機能を持つ。UvicornはFastAPIの非同期性能を最大限に引き出すための標準的なサーバーであり、この組み合わせはコンテナ環境での軽量なAPIサービス開発に適している。

Infrastructure as Code (IaC)
Terraform:
用途: AWS上に必要なインフラストラクチャ（ECRリポジトリ, EC2インスタンス, IAMロール, セキュリティグループ）をコードで定義し、自動で構築・管理するために使用する。
選定理由: クラウドプロバイダーに依存しない宣言的な構文でインフラを記述できる、IaCのデファクトスタンダードである。手動でのコンソール操作を排除し、インフラの構築プロセスを再現可能でバージョン管理可能なものにするため採用した。

Amazon Web Services (AWS):
用途: アプリケーションを実行するためのクラウドインフラストラクチャを提供する。EC2（仮想サーバー）、ECR（コンテナレジストリ）、IAM（権限管理）、VPC（ネットワーク）を利用する。
選定理由: 幅広いサービスと高い信頼性を持つ主要なクラウドプロバイダーであり、Terraformとの連携も成熟しているため採用した。

CI/CDとコンテナ化
Docker:
用途: FastAPIアプリケーションとその依存関係を、自己完結したコンテナイメージとしてパッケージングする。
選定理由: アプリケーションの実行環境をコード化（Dockerfile）することで、開発から本番まで一貫した環境を保証し、「どこでも同じように動く」ことを実現する。現代的なアプリケーションデプロイにおける標準技術である。

GitHub Actions:
用途: CI/CDパイプラインを定義・実行するプラットフォーム。コードの変更をトリガーに、テストの実行、Dockerイメージのビルド、そしてAmazon ECRへのプッシュを自動化する。
選定理由: GitHubにネイティブに統合されており、リポジトリの設定ファイル（.github/workflows/）でパイプラインをコードとして管理できる。テストとデプロイのプロセスを自動化することで、迅速かつ信頼性の高いリリースサイクルを実現するため採用した。

2. 分析モデル（分析手法）の選択
このプロジェクトは特定の分析モデルを扱うものではなく、現代的なソフトウェア開発・運用のプラクティスである Infrastructure as Code (IaC) と CI/CD を実装することに主眼を置いている。

Infrastructure as Code (IaC) によるインフラ管理:
手法: main.tf をはじめとするTerraformの設定ファイルに、必要なAWSリソースとその設定（例: セキュリティグループのポート開放ルール、EC2インスタンスのタイプ）を宣言的に記述する。terraform apply コマンドを実行すると、Terraformが現在のAWSの状態とコードの記述を比較し、差分を自動的に適用してインフラを構築・変更する。
選択理由: 手動でのインフラ構築に伴うヒューマンエラーや設定漏れのリスクを排除し、インフラの構成をコードとしてバージョン管理できる。これにより、誰が実行しても同じ構成のインフラを正確かつ迅速に再現することが可能になる。

CI/CDパイプラインによるデプロイ自動化:
手法: ci_cd_pipeline.yml に定義されたGitHub Actionsのワークフローに基づき、以下のプロセスを自動実行する。
  ・継続的インテグレーション (CI): 開発者がコードをプッシュすると、自動的にテストが実行され、コードの品質が一定基準を満たしているか（リグレッションが発生していないか）を検証する。
  ・継続的デリバリー (CD): テストが成功したコードを、Dockerfile に基づいてDockerイメージとしてビルドし、Amazon ECR（コンテナレジストリ）にプッシュする。
選択理由: アプリケーションのテストからデプロイ可能な成果物（Dockerイメージ）の作成までを自動化することで、開発サイクルを高速化し、手作業によるミスをなくす。これにより、開発者はコードの品質向上に集中でき、常にデプロイ可能な状態が維持される。

3. プログラム全体の流れ（アーキテクチャ）
このプロジェクトは、物理的なインフラと、その上で動作するアプリケーションを、2つの独立しつつも連携する自動化フローによって構築・デプロイするアーキテクチャを採用している。
フロー1: インフラストラクチャのプロビジョニング (Terraform)
責務: アプリケーションを実行するためのAWS上の基盤を構築する。
流れ:
  ・開発者がローカル環境で terraform apply を実行する。
  ・Terraformは main.tf の定義に従い、AWS上に以下のリソースを順番に作成する。
    ・aws_ecr_repository: アプリケーションのDockerイメージを保存するためのプライベートなリポジトリ。
    ・aws_iam_role: EC2インスタンスがECRからイメージを読み取るための権限を定義する。
    ・aws_security_group: EC2インスタンスへのHTTP（80番ポート）アクセスを許可するファイアウォールルール。
    ・aws_instance: アプリケーションをホストするEC2インスタンス本体。
EC2インスタンスの起動時、user_data に指定された user_data.sh スクリプトが自動的に実行される。このスクリプトは、Dockerのインストール、ECRへのログイン、そしてECRから :latest タグの付いたアプリケーションイメージをプルしてコンテナとして起動する役割を持つ。Terraformの templatefile 関数により、ECRリポジトリのURLがスクリプトに動的に埋め込まれる。

フロー2: アプリケーションのビルドとデプロイ (GitHub Actions)
責務: FastAPIアプリケーションのコードをテストし、コンテナ化してECRに配置する。
流れ:
  ・開発者が main ブランチにコードをプッシュする。
  ・GitHub Actionsがこれをトリガーし、ci_cd_pipeline.yml で定義されたワークフローを開始する。
  ・testジョブ: Pytestによる自動テストが実行される。
  ・build-and-pushジョブ: testジョブが成功すると、このジョブが開始される。
    ・AWSの認証情報（Secretsとして保存）を使い、ECRにログインする。
    ・Dockerfile に基づいてFastAPIアプリケーションをコンテナイメージとしてビルドする。
    ・ビルドされたイメージに :latest タグを付け、フロー1で作成されたECRリポジトリにプッシュする。

全体の連携:
この2つのフローにより、インフラの構築とアプリケーションのデプロイが分離・自動化される。一度Terraformでインフラを構築すれば、以降は開発者がGitHubにコードをプッシュするだけで、テスト済みの最新アプリケーションがECRに自動的に配置される。
